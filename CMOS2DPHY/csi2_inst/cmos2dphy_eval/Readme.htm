CMOS to MIPI D-PHY Interface Bridge Soft IP

IP Generation through Lattice Diamond


Generating the IP in Clarity Designer

Clarity Designer is a tool to customize modules and IPs and place them into the device’s architecture. Besides configuration and generation of modules and IPs, Clarity Designer can also create a top module template in which all generated modules and IPs are instantiated.

 

To configure the IP in Clarity Designer :

  1. Input valid values in the required fields in the Configuration tab.
    • Number of TX Channels : [Read-Only] Specify the number of TX channels. Default is set to 1. Only 1 TX channel is supported in LIFMD device.
    • Number of TX Lanes : [User-Input] Specify the number of TX lanes. 1, 2, 4 lanes are supported.
    • TX Interface : [User-Input] Specify the interface for the TX. DSI, CSI2 are supported.
    • TX Gear : [Read-Only] Specify the gearing for the TX. Only gear 8 is supported in this IP.
    • RX Line Rate : [Read-Only] Specify the RX line rate. Automatically computed based on the desired TX line rate.
    • TX Line Rate : [Read-Only] Specify the total target TX line rate. Expected to be the same with RX line rate.
    • TX Line Rate (per lane) : [User-Input] Specify the target TX line rate per lane. All other clocks are computed based on this input.
    • DPHY Clock Frequency : [Read-Only] Specify the DPHY clock frequency. Automatically computed based on the desired TX line rate.
    • DPHY Clock Mode : [User-Input] Specify the TX clock mode.
    • Byte Clock Frequency : [Read-Only] Specify the internal byte clock frequency. Automatically computed on from the desired TX line rate.
    • Pixel Clock Frequency : [Read-Only] Specify the pixel clock frequency. Automatically computed based on the desired TX line rate.
    • DCS Clock Frequency : [Read-Only] Specify the DCS clock frequency. Automatically computed based on the desired TX line rate. Used when DCS is transmitted in High Speed mode.
    • tINIT_SLAVE Value : [User-Input] Specify delay time for TX D-PHY TINIT requirement. Must satisfy TX D-PHY TINIT minimum requirement of 100 µs and is clocked by byte clock.
    • Enable miscellaneous status signals : [User-Input] Click to enable the status signals (i.e. PLL lock)
    • Bypass TINIT : [User-Input] Click to disable the delay for initialization counter.
  2. Input valid values in the required fields in the Video tab.
    • Data Type : [User-Input] Specify the data type depending on the TX interface selected.
    • Virtual Channel : [User-Input] Specify the Virtual Channel ID.
    • Word Count : [User-Input] Specify the number of bytes within the valid payload.
    • Number of DCS words : [User-Input] Specify the number of DCS words for the display if DSI is selected.
    • DCS ROM Wait Time : [User-Input] Specify delay time for sending DCS commands. This is clocked by byte clock if DCS is in High Speed mode and DCS clock when DCS is in Low Speed mode. This needs to be adjusted if DCS is not meeting protocol timing requirements. Normal value is 1200 based on 111.375 MHz byte clock.
    • DCS Mode : [User-Input] Specify the DCS transmission mode: High-Speed or Low-Speed
    • DCS ROM Initialization File : [User-Input] Specify the file which contains the DCS commands to be sent to the display. Refer to the UserGuide document for the proper format.
    • Bypass DCS : [User-Input] Click to disable the DCS transmission for the TX channel.
  3. Go to Protocol Timing Parameters tab. Input valid values in the required fields if customization is desired.
    • Data HS-Prepare : [User-Input] Specify the time that the DPHY TX drives the Data Lane LP-00 Line state immediately before the HS-0 Line state starting the HS transmission. Timing parameter is in number of byte clock cycles. Value is initially automatically computed but can be customized.
    • Data HS-Zero : [User-Input] Specify the time that the DPHY TX drives the HS-0 state prior to transmitting the Sync sequence. Timing parameter is in number of byte clock cycles. Value is initially automatically computed but can be customized.
    • Clock Pre : [User-Input] Specify the time that the HS clock shall be driven by the DPHY TX prior to any associated Data Lane beginning the transition from LP to HS mode. Timing parameter is in number of byte clock cycles. Value is initially automatically computed but can be customized.
    • Clock Post : [User-Input] Specify the time that the DPHY TX continues to send HS clock after the last associated Data Lane has transitioned to LP Mode. Interval is defined as the period from the end of THS-TRAIL to the beginning of TCLK-TRAIL. Timing parameter is in number of byte clock cycles. Value is initially automatically computed but can be customized.
  4. Click the Configure button after the required parameters are selected.
  5. Click Close. The IP will be generated.
 

Implementing the IP Module Using Lattice Diamond


Instantiating the IP

The core modules of CMOS to MIPI D-PHY Interface Bridge Soft IP are synthesized and provided in NGO format with black box Verilog source files for synthesis. A Verilog source file named , <instance_name>_cmos_2_dphy_ip.v instantiates the black box of core modules. The top-level file <instance_name>.v instantiates <instance_name>_cmos_2_dphy_ip.v and DCS components. A Verilog instance template <instance_name>_inst.v or VHDL instance template <instance_name>_inst.vhd is also provided as a guide for users who want to instantiate the generated soft IP top-level in their own top-level module.

The user does not need to instantiate the IP instances one by one manually. The top-level file along with the other Verilog source files are provided in \<project_dir>. These files are refreshed each time the IP is regenerated.


Hardware Evaluation

Lattice's IP hardware evaluation capability makes it possible to create IP cores that operate in hardware for a limited period of time (approximately four hours) without requiring the request on an IP license. The hardware evaluation capability is enabled by default. It can be disabled by Project->Active Strategy->Translate Design Settings. The setting is called "Hardware Evaluation" and the options are "Enable" or "Disable".
 
When the Hardware Evaluation feature is enabled in the design, it will generate a programming file that may be downloaded into the device. After initialization, the IP will be operational for approximately four hours. After four hours, the device will stop working and it will be necessary to reprogram the device to re-enable operation. This hardware evaluation capability is only enabled if the core has not been licensed. During implementation, a license check is performed. If the hardware evaluation feature is disabled, a pop-up window will be displayed indicating a license failure. Click"OK" in the window and the bitstream will not be generated. If a license is detected, no pop-up window is displayed and core generation is completed with no restrictions.


Implementing the IP in a Top-Level Design


In Clarity Designer, the Clarity Designer project file (.sbx) is added to Lattice Diamond as a source file after all IPs are generated. Note that default Diamond strategy (.sty) and default Diamond preference file (.lpf) will be used. When using the .sbx approach, import the recommended strategy and preferences from <\project_dir>\cmos2dphy_eval\<instance_name>\impl\lifmd\< synplify|lse> directories. All required files are invoked automatically. You can directly synthesize, map and place/par the design in the Diamond design environment after the cores are generated.

Push-button implementation of this top-level design with either Synplify or Lattice Synthesis Engine is supported via the Diamond project files <instance_name>_top.ldf located in <project_dir>\cmos2dphy_eval\<instance_name>\impl\lifmd\<synplify|lse>.

 

To use the project file in Diamond:

  1. Choose File > Open > Project under the File tab in Diamond.
  2. Browse to <\project_dir>\cmos2dphy_eval\<instance_name>\impl\lifmd\<synplify|lse>in the Open Project dialog box.
  3. Select and open <instance_name_top>. ldf. At this point, all of the files needed to support top-level synthesis and implementation will be imported to the project.
  4. One lpf file is available in the \impl\lifmd\<synplify|lse> directory <instance_name>_top.lpf.
  5. Select the Process tab in the left-hand GUI window.
  6. Implement the complete design via the standard Diamond GUI flow.

Running Functional Simulation

To run simulation using Active HDL:

  1. Modify the "do" file located in \<project_dir>\cmos2dphy_eval\<instance_name>\sim\aldec\
    1. Specify working directory (sim_working_folder).
       i.e. set sim_working_folder "C:/my_design"
    2. Specify workspace name that will be created in working directory.
       i.e. set workspace_name "design_space"
    3. Specify design name.
       i.e. set design_name "DesignA"
    4. Specify design path where the IP Core generated using Clarity Designer is located.
       i.e. set design_path "C:/my_designs/DesignA"
    5. Specify design instance name (same as the instance name specified in Clarity Designer).
       i.e. set design_inst "DesignA_inst"
    6. Specify Lattice Diamond Primitive path (diamond_dir) to where it is installed.
       i.e. set diamond_dir "C:/lscc/diamond/3.8_x64"
    7. Update testbench parameters to customize data size, clock and/or other settings. See additional info related to Testbench Compiler Directives below.
  2. Under the Tools Menu, select Active-HDL.
  3. In Active-HDL window, under the Tools tab, select Execute Macro.
  4. Select the "do" file <\project_dir>\cmos2dphy_eval\<instance_name>\sim\aldec\*.do
  5. Select OK.
  6. Wait for simulation to finish.
  7. Below are the testbench directives which can be modified by setting the define in the vlog command in the "do" file.

    Example:
    vlog \
    +define+NUM_FRAMES=60 \
    +define+NUM_LINES=1080 \
    ....
    * NUM_PIXELS : Number of pixels per line
    * NUM_LINES : Number of lines per frame
    * NUM_FRAMES : Number of frames to be transmitted
    * PIX_CLK : clock period in ps; used as reference clock
    *These parameters can be manually included in a "do" file or TB define file.

    Optional:
    CSI2 parameters
    * INIT_DRIVE_DELAY : delay from reset deassertion or tinit_done assertion before start of transaction
    * FV_H_TO_LV_H : used to inject delay (in terms of clock cycle) from fv assertion to lv assertion
    * LV_L_TO_LV_H : used to inject delay (in terms of clock cycle) from lv negation to lv assertion
    * LV_L_TO_FV_L : used to inject delay (in terms of clock cycle) from lv negation to fv negation
    * FV_L_TO_FV_H : used to inject delay (in terms of clock cycle) from fv negation to fv assertion

    DSI parameters
    * INIT_DRIVE_DELAY : delay from reset deassertion or tinit_done assertion before start of transaction
    * HFRONT_PORCH : used to set duration of HFRONT_PORCH in terms of clock cycles
    * HSYNC_PULSE : used to set duration of HSYNC_PULSE in terms of clock cycles
    * HBACK_PORCH : used to set duration of HBACK_PORCH in terms of clock cycles
    * VFRONT_PORCH : used to set duration of VFRONT_PORCH in terms of hsync pulses
    * VSYNC_PULSE : used to set duration of VSYNC_PULSE in terms of hsync pulses
    * VBACK_PORCH : used to set duration of VBACK_PORCH in terms of hsync pulses
    * User can override the default timing parameters using defines above.

     Refer to UserGuide document for more information.
